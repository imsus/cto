Risks & Recommendations
⚠ Bus Factor = 2
Only two developers own the entire system. If either leaves, ~50% of institutional knowledge is gone. No code review overlap between them — each domain has a single point of failure.
🔧 Jira Tracking Gap
The earlier Jira export ("CCID H2 2025") only showed Imam as assignee, making Dewo's contributions invisible to management. Proper assignee tracking is essential for accurate performance reviews and resource planning.
📊 Backend Documentation Debt
Imam owns architecture decisions (data models, API schemas, infra config) that are primarily in his head. Recommend API docs, runbooks, and architecture decision records before scaling the team.
💡 Scaling Recommendation
Adding a 3rd developer? Target the frontend side — Dewo has the bigger ticket volume (66 vs 56) and more user-facing features. A second frontend dev would unblock the faster-growing surface area.
Bottom Line for Leadership
Two developers built a full influencer marketing platform (CMS + WhatsApp chat + content drafting + data pipeline) in 6 months, from zero to production. The team split is effective — Imam owns the foundation (servers, databases, APIs), Dewo owns the experience (screens, reports, chat). Both carry roughly equal workload (47%/53% by SP). The biggest risk is knowledge concentration — every domain has exactly one owner. Mitigate with documentation and cross-training before the next hire.
